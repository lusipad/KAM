## 1. 产品概述

### 1.1 产品定位
AI工作助手是一款面向个人开发者和小型研发团队的工作助手系统，定位是“人的外置大脑 + Agent 指挥台”。它负责承接任务、沉淀上下文、组织笔记、并发调度外部 Agent（如 Codex、Claude Code）执行具体工作，并将结果、diff、日志和结论统一收口，帮助人减少琐碎事务和上下文切换成本。

### 1.2 目标用户
- **个人开发者**: 需要统一管理任务、想法、资料和多个 AI 工具的执行结果。
- **小团队成员**: 需要在 ADO/Git/文档之间快速切换并获取当前任务上下文。
- **Tech Lead / 架构师**: 需要把调研、实现、评审、文档等任务拆给不同 Agent 并统一收口。

### 1.3 核心价值主张
| 价值维度 | 描述 |
|----------|------|
| **任务外置化** | 把脑内待办、想法、链接、需求和上下文沉淀到系统里，而不是散落在聊天、便签和浏览器标签页中 |
| **统一上下文** | 把 ADO、代码仓库、笔记、文档和历史执行结果串起来，为每次任务自动组装上下文 |
| **并发 Agent 调度** | 一个任务可拆给多个 Agent 并发执行，适合调研、编码、文档、评审等并行工作 |
| **结果统一收口** | 统一查看 diff、日志、结论、风险说明和后续建议，降低多 Agent 输出的管理成本 |
| **轻量可落地** | 以开发内用为前提，尽量复用成熟工具，不以做成企业级平台为目标 |

### 1.4 设计原则
| 原则 | 说明 |
|------|------|
| **人主导而非 Agent 自治** | 人负责给目标、看过程、做最终判断，Agent 负责具体执行 |
| **最小可用优先** | 先做任务台、上下文、并发调度、结果收口，不做大而全平台 |
| **外部工具复用** | Codex、Claude Code、ADO、Git 等继续做各自擅长的事，本系统只做调度和整合 |
| **文本与链接优先** | 先依赖任务卡、笔记、引用和全文检索，不强依赖图谱或复杂知识建模 |
| **默认轻量记忆** | 先保存笔记、任务历史、上下文快照和常用提示，长期记忆治理延后 |
| **先本地可用再平台扩展** | 优先支持本地/小团队使用，复杂治理能力后置 |

### 1.5 产品边界
| 平台负责 | 平台不负责 |
|----------|------------|
| 任务收件、笔记沉淀、上下文组装、Agent 调度、结果收口 | 取代 IDE、ADO、Git、Wiki 成为新的主工作系统 |
| 管理多 Agent 的运行记录、产物和状态 | 让 Agent 自主协商并长期自治而不受人控制 |
| 为开发工作提供轻量助手能力 | 一开始就做成企业级治理平台、审批平台或低代码平台 |

## 2. 核心功能模块

### 2.0 P0 交付策略
Lite 版本的优先级按“是否能成为日常工作的外置大脑”定义，而不是按“模块是否完整”定义。P0 场景建议如下：

| 场景 | 输入 | 输出 | 价值 |
|------|------|------|------|
| **收件箱整理到任务卡** | 想法、链接、截图、需求、会议结论 | 可执行任务卡、优先级、上下文引用 | 降低脑内负担 |
| **项目问答与上下文打包** | ADO 工作项、代码仓库、设计文档、历史笔记 | 简洁答案、来源引用、上下文快照 | 减少查资料时间 |
| **并发 Agent 执行** | 任务描述、代码仓库、约束条件 | 多个 Agent 的执行结果、日志、diff、总结 | 提高并行工作效率 |
| **结果收口与评审** | Agent 输出、命令日志、修改结果 | 汇总结论、风险项、下一步建议 | 降低多 Agent 管理复杂度 |
| **开发状态摘要** | 工作项、PR、构建状态、手动记录 | 日报/周报/当前阻塞摘要 | 减少重复汇总工作 |

上述场景由任务台、轻量记忆、上下文服务、Agent 指挥台和开发系统集成共同承载，重点是“把工作接住并推进”，而不是做复杂平台能力展示。

### 2.1 收件箱与工作笔记模块 (Inbox & Notes)

#### 功能概览
系统的第一入口，用来承接任务、灵感、资料和过程记录。它不是完整的知识管理平台，而是面向开发工作流的轻量笔记与任务收件箱。

#### 核心功能

| 功能 | 优先级 | 描述 |
|------|--------|------|
| **快速收件** | P0 | 随手记录任务、想法、会议结论、网址和文件引用 |
| **Markdown笔记** | P0 | 支持任务说明、方案草稿、Agent 提示和复盘记录 |
| **任务卡管理** | P0 | 为每个任务保存标题、状态、优先级、来源和关联资源 |
| **标签与引用** | P0 | 支持标签、项目关联、文件链接和工作项链接 |
| **全文搜索** | P0 | 按标题、正文、标签和引用内容搜索 |
| **上下文固定** | P1 | 将常用文档、仓库路径、工作项、Prompt 固定到任务 |
| **摘要生成** | P1 | 对笔记、会议记录或长文本生成简要摘要 |
| **导入导出** | P2 | 导入 Markdown、文本、会议纪要，导出任务总结 |

#### 技术实现要点
- **编辑器**: Tiptap / Markdown 编辑器
- **搜索**: PostgreSQL 全文搜索优先
- **数据模型**: Notes + Tasks + Links
- **文件引用**: 先支持路径、URL、工作项 ID 等轻量引用方式

#### 数据模型
```typescript
interface TaskCard {
  id: UUID;
  title: string;
  description: string;
  status: 'inbox' | 'ready' | 'running' | 'review' | 'done';
  priority: 'low' | 'medium' | 'high';
  sourceRefs: string[];
  noteId?: UUID;
  assigneeAgents: string[];
  tags: string[];
  createdAt: DateTime;
  updatedAt: DateTime;
}
```

### 2.2 轻量记忆与上下文模块 (Context & Memory)

#### 功能概览
Lite 版本不追求复杂长期记忆体系，而是围绕“当前任务怎么更快拿到可用上下文”设计。记忆的主要载体是任务历史、笔记、对话摘要、固定引用和上下文快照。

#### 核心功能

| 功能 | 描述 |
|------|------|
| **上下文打包** | 根据任务自动聚合 ADO、代码、文档、笔记和历史输出 |
| **对话摘要** | 将长对话压缩为可复用摘要，减少后续重复输入 |
| **固定上下文** | 允许用户手动 pin 住关键文件、工作项、仓库目录和提示词 |
| **历史检索** | 检索过去任务、运行结果、复盘和常用模板 |
| **可选语义检索** | 当全文搜索不够时，再为文档问答场景补充向量检索 |

#### 技术实现要点
- **主检索方式**: PostgreSQL 全文搜索 + 标签/引用过滤
- **上下文缓存**: Redis / 本地缓存
- **向量能力**: P1/P2 可选启用 `pgvector`
- **存储对象**: context snapshots、history summaries、task artifacts

### 2.3 Agent 指挥台模块 (Agent Command Center)

#### 功能概览
Agent 指挥台是 Lite 版本的核心。它负责把任务交给外部 Agent 执行，并统一管理运行状态、上下文、日志、产物和结果。重点不是设计复杂代理角色，而是让 Codex、Claude Code 这类现成 Agent 成为可调度 worker。

#### Agent 类型

| 类型 | 示例 | 主要职责 |
|------|------|----------|
| **Coding Agent** | Codex, Claude Code | 编码、重构、修复、测试、生成 patch |
| **Research Agent** | 通用 LLM / CLI 工具链 | 调研方案、整理资料、汇总差异 |
| **Review Agent** | 通用 LLM / 自定义脚本 | 审查结果、提炼风险、生成 review note |

#### 核心功能

| 功能 | 描述 |
|------|------|
| **任务派发** | 将任务按类型派发给一个或多个 Agent 执行 |
| **并发运行** | 支持多个 Agent 同时执行不同任务或同一任务的不同子任务 |
| **隔离工作目录** | 为每个 Agent 分配独立 worktree / 工作目录，避免互相覆盖 |
| **运行追踪** | 跟踪开始时间、状态、日志、结果摘要和失败原因 |
| **产物收集** | 汇总 diff、patch、文档草稿、命令输出和建议 |
| **人工收口** | 人可以中断、重跑、比较不同 Agent 结果并做最终决策 |

#### 技术实现要点
- **Agent 适配方式**: CLI adapter / subprocess
- **工作区隔离**: git worktree + 独立运行目录
- **调度框架**: 轻量任务队列，必要时再引入 LangGraph
- **运行协议**: 统一 run metadata、artifact metadata、status events

### 2.4 开发系统集成模块 (ADO / Git / Docs)

#### 功能概览
Azure DevOps、Git 仓库和项目文档仍然是事实源。Lite 版本不追求复杂双向同步和审批写回，而是优先做好读取、引用和上下文增强，让 Agent 能基于最新项目状态工作。

#### 集成功能

| 模块 | 核心功能 |
|------|----------|
| **只读查询** | WIQL 查询、工作项详情、仓库/PR、构建状态、文件内容 |
| **上下文引用** | 将工作项、PR、提交、文档链接固定到任务上下文 |
| **状态摘要** | 汇总项目进度、阻塞项、待处理事项和构建情况 |
| **手动写回** | 先生成评论草稿或工作项草稿，再由人决定是否写回 |
| **轻同步** | 手动刷新或低频增量同步，不做复杂事件总线 |

#### 身份验证方案

| 方式 | 适用场景 | 推荐度 |
|------|----------|--------|
| **PAT** | Azure DevOps Server/自动化脚本 | ⭐⭐⭐⭐ |
| **OAuth 2.0** | Azure DevOps Services/第三方应用 | ⭐⭐⭐⭐⭐ |
| **NTLM/Kerberos** | 企业内网/域环境 | ⭐⭐⭐⭐ |

#### 数据同步策略

```
┌─────────────────────────────────────────────────────────┐
│                    Lite 数据同步架构                      │
├─────────────────────────────────────────────────────────┤
│  手动刷新                                               │
│  • 当前任务需要时主动获取最新数据                        │
├─────────────────────────────────────────────────────────┤
│  低频定时同步                                           │
│  • 工作项/PR/构建状态按需每15-30分钟同步                │
├─────────────────────────────────────────────────────────┤
│  本地上下文缓存                                         │
│  • 缓存最近访问的工作项、文档和仓库元数据               │
└─────────────────────────────────────────────────────────┘
```

#### 技术实现要点
- **API客户端**: HttpClient / 官方 SDK
- **认证方式**: PAT 优先
- **缓存方式**: 最近任务上下文缓存
- **写回策略**: 默认人工确认，不做自动落地

### 3.2 模块间交互

#### 交互关系

| 交互 | 描述 |
|------|------|
| **外部系统 → Context Builder** | 通过 ADO/Git/文档连接器按需读取数据，组装当前任务上下文 |
| **Tasks ↔ Notes** | 任务卡与笔记互相关联，形成任务说明、过程记录和复盘沉淀 |
| **Task Service ↔ Agent Runner** | 任务服务决定派发哪个 Agent、是否并发运行、是否需要重新执行 |
| **Agent Runner ↔ Review Service** | Agent 输出的 patch、日志和摘要进入评审中心统一收口 |
| **Conversation ↔ Context Builder** | 对话入口使用任务上下文和历史摘要增强回答质量 |

### 3.3 数据流设计

```
1. 用户把任务、想法和链接放入收件箱
   - 手动输入、复制链接、导入会议结论、引用工作项
2. Context Builder 组装任务上下文
   - 读取 ADO、仓库、笔记、历史运行记录和固定引用
3. Agent Runner 派发任务
   - 选择 Codex、Claude Code 或其他 Agent，并发执行
4. Review Service 收口结果
   - 汇总 diff、日志、总结、风险项和后续建议
5. 用户做最终判断
   - 采纳结果、继续追问、重跑某个 Agent、手动写回外部系统
```

## 4. 技术栈建议

### 4.1 整体技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **后端** | Python + FastAPI | 轻量 API 与 Agent 调度服务 |

## 5. 实施路线图

### 5.1 阶段规划

Lite 版本的路线图按“是否真的成为开发工作的外置大脑”来定义，而不是按模块覆盖率定义。

| 阶段 | 建议周期 | 目标 | 关键交付 |
|------|----------|------|----------|
| **Phase 0: 单人可用底座** | 2-3周 | 先把任务、笔记和项目上下文接住 | 收件箱、任务卡、Markdown 笔记、ADO/Git 基础读取、全文搜索 |
| **Phase 1: Agent 指挥台** | 3-4周 | 让多个 Agent 真正并发工作 | Codex / Claude Code 任务派发、运行状态、日志、diff、结果收口 |
| **Phase 2: 上下文增强** | 3-4周 | 让系统更像外置大脑 | 上下文快照、历史任务检索、对话摘要、固定引用、状态摘要 |
| **Phase 3: 可选增强** | 持续 | 在真实使用痛点出现后补充能力 | 向量检索、模板库、更多连接器、轻量权限、多用户协作 |

### 5.2 详细时间线

| 阶段 | 时长 | 核心交付 | 验收里程碑 |
|------|------|----------|------------|
| **Phase 0** | 2-3周 | 任务台与笔记底座 | 能记录任务、搜索笔记、查看 ADO/Git 基础信息 |
| **Phase 1** | 3-4周 | 并发 Agent 调度 | 至少 2 类 Agent 可并发运行，结果可统一查看和比较 |
| **Phase 2** | 3-4周 | 上下文与历史复用 | 系统能基于历史任务和项目资料自动组装上下文 |
| **Phase 3** | 持续 | 可选增强项 | 明确出现检索不足、协作不足或权限需求时再补能力 |

### 5.3 团队配置建议

| 角色 | Phase 0-1 | Phase 2 | Phase 3 |
|------|-----------|---------|---------|
| 全栈/后端工程师 | 1 | 1-2 | 1-2 |
| 前端工程师 | 1 | 1 | 1 |
| AI应用工程师 | 0-1 | 1 | 1 |
| 产品/使用者本人 | 1 | 1 | 1 |
| **总计** | **2-4** | **3-5** | **3-5** |

建议由非常小的团队直接围绕真实使用场景迭代，而不是按组织化平台项目立项。

## 6. 风险评估与应对

### 6.1 技术风险

| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| **AI成本控制** | 高 | 高 | 分层模型、语义缓存、Token优化 |
| **向量检索准确性** | 高 | 中 | 混合检索、重排序、持续调优 |
| **Azure DevOps API限流** | 中 | 高 | 增量同步、批量处理、缓存策略 |
| **大规模性能** | 高 | 中 | 分页、索引优化、缓存、CDN |

### 6.2 业务风险

| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| **用户接受度** | 高 | 中 | 渐进式功能、用户反馈循环 |
| **数据安全** | 高 | 低 | 加密、审计、合规认证 |
| **竞品压力** | 中 | 高 | 差异化功能、垂直深耕 |

### 6.3 非功能性需求

| 维度 | 目标 | 设计要求 |
|------|------|----------|
| **响应速度** | 打开任务和查看上下文要足够快；长任务异步执行 | 常用数据缓存、任务异步化、避免过度检索 |
| **运行可靠性** | Agent 任务失败后可重试、可中断、可重新派发 | 任务状态机、失败原因记录、手动重跑 |
| **隔离性** | 多 Agent 并发时不互相污染工作区 | git worktree / 独立目录 / 独立日志 |
| **可追踪性** | 每次任务的上下文、命令、输出、diff 和结论都可回看 | run logs、artifact store、history timeline |
| **简洁性** | 不引入明显超出当前需求的复杂组件 | 单库优先、少服务优先、配置最小化 |
| **可扩展性** | 后续可补更多 Agent 和检索能力，但不影响当前结构 | adapter 模式、标准 run metadata、可插拔连接器 |
