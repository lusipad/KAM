## 1. 产品概述

### 1.1 产品定位
AI工作助手是一款面向研发团队和知识工作者的工作流型上下文平台。它不替代 Azure DevOps、Git、Wiki、文档库等现有业务系统，而是在这些系统之上建立统一的上下文层、检索层和自动化执行层，通过模板化 AI 工作流提升团队效率。

### 1.2 目标用户
- **团队负责人/Tech Lead/PM**: 需要把需求、项目状态和知识资产串联起来，降低跨系统协调成本。
- **一线执行人员**: 需要快速获得项目上下文、生成草稿、补全文档、汇总状态。
- **平台与架构团队**: 需要可复用、可治理、可审计的 AI 能力底座。

### 1.3 核心价值主张
| 价值维度 | 描述 |
|----------|------|
| **统一上下文** | 连接知识库、对话、项目系统和历史记录，形成可检索、可追溯的团队上下文层 |
| **工作流自动化** | 围绕高频场景提供模板化闭环，如需求拆解、周报汇总、会议纪要、发布检查 |
| **可控执行** | 默认读优先，写操作需审批、可回滚、可追踪，不把系统变成不可控自动化黑盒 |
| **生产级治理** | 从一开始纳入权限、审计、评测、成本控制、提示词版本化和连接器治理 |

### 1.4 设计原则
| 原则 | 说明 |
|------|------|
| **工作流优先** | 优先交付能直接节省团队时间的高频闭环，而不是堆叠并列模块 |
| **外部系统为事实源** | ADO、代码仓库、知识库等仍是 source of truth，本平台负责聚合、分析和受控写回 |
| **代理能力内化** | ClawTeam 作为内部执行引擎保留，普通用户默认使用模板化工作流，不直接编排多代理 |
| **记忆可治理** | 记忆必须具备类型、来源、置信度、过期策略和人工修正能力 |
| **读优先写受控** | 默认优先检索、总结、建议；涉及创建或修改外部系统数据时必须经过审批或显式确认 |
| **生产治理前置** | 安全、合规、审计、评测、观测性不是上线后补充项，而是架构前提 |

### 1.5 产品边界
| 平台负责 | 平台不负责 |
|----------|------------|
| 统一检索、上下文组装、模板化 AI 工作流、受控写回、执行追踪 | 取代 Azure DevOps / Git / Wiki 成为新的主业务系统 |
| 为团队提供可解释、可审计、可复用的自动化能力 | 默认自动改写外部系统数据而不留痕 |
| 把代理、记忆、知识库等能力收敛成稳定产品能力 | 让普通用户直接配置复杂多代理拓扑和底层模型策略 |

## 2. 核心功能模块

### 2.0 P0 交付策略
优先级不再按“模块是否存在”定义，而按“是否形成高频工作流闭环”定义。P0 场景建议如下：

| 场景 | 输入 | 输出 | 价值 |
|------|------|------|------|
| **需求拆解到工作项** | PRD/需求说明/会议结论 | 工作项草稿、拆解建议、负责人建议 | 缩短需求落地时间 |
| **迭代日报/周报生成** | ADO工作项、构建状态、PR活动 | 结构化状态摘要、风险项、阻塞项 | 减少手工汇总 |
| **新人项目问答** | 项目文档、历史对话、工作项 | 可追溯答案和来源引用 | 降低 onboarding 成本 |
| **会议纪要转知识与行动项** | 会议文本/录音转写 | 笔记、待办草稿、关联项目上下文 | 提升记录复用率 |
| **发布风险检查** | 构建状态、待发布工作项、历史事故记忆 | 风险清单、检查项、建议动作 | 降低发布失误 |

上述场景由知识管理、长期记忆、工作流引擎、外部集成和对话界面共同承载，模块只是能力载体，不是最终交付物。

### 2.1 知识管理模块 (Knowledge Management)

#### 功能概览
类似Obsidian + NotebookLM的知识记录学习系统，支持双链笔记、AI辅助整理和语义搜索。

#### 核心功能

| 功能 | 优先级 | 描述 |
|------|--------|------|
| **Markdown编辑器** | P0 | 支持标准Markdown语法，实时预览 |
| **富文本编辑器** | P0 | WYSIWYG编辑，适合非技术用户 |
| **双链笔记系统** | P0 | Wiki链接`[[Note]]`、标签`#tag`、块引用`^block-id` |
| **知识图谱视图** | P1 | 全局/局部图谱、路径发现、聚类分析 |
| **AI自动标签** | P1 | 保存时自动提取标签 |
| **智能摘要** | P1 | AI生成文档摘要 |
| **关联建议** | P1 | 编辑时推荐相关笔记 |
| **语义搜索** | P1 | 向量语义检索 + 全文检索混合 |
| **导入导出** | P2 | Obsidian、Notion、PDF等格式 |

#### 技术实现要点
- **编辑器**: ProseMirror + Tiptap (支持Markdown/富文本/块编辑三模式)
- **图谱可视化**: D3.js / react-force-graph
- **向量存储**: PostgreSQL + pgvector
- **全文搜索**: Meilisearch
- **协同编辑**: Yjs + WebSocket

#### 数据模型
```typescript
interface Note {
  id: UUID;
  title: string;
  content: string;
  contentType: 'markdown' | 'rich-text' | 'block';
  parentId?: UUID;
  path: string;
  version: number;
  metadata: {
    tags: string[];
    aliases: string[];
    extractedTags: string[];
    summary?: string;
    properties: Record<string, any>;
  };
  stats: {
    wordCount: number;
    linkCount: number;
    backlinkCount: number;
    viewCount: number;
  };
  createdAt: DateTime;
  updatedAt: DateTime;
}
```

### 2.2 长期记忆模块 (Long-term Memory)

#### 功能概览
三层记忆架构，让AI记住用户的工作习惯、偏好和历史上下文。

#### 记忆分层

```
┌─────────────────────────────────────────────────────────┐
│                    记忆系统架构                          │
├─────────────────────────────────────────────────────────┤
│  工作记忆层 (Working Memory)                            │
│  • 当前会话上下文 | 容量: 4-7个信息块 | 保留: 会话期间  │
├─────────────────────────────────────────────────────────┤
│  短期记忆层 (Short-term Memory)                         │
│  • 近期对话历史 | 容量: 最近N轮对话 | 保留: 1-7天       │
├─────────────────────────────────────────────────────────┤
│  长期记忆层 (Long-term Memory)                          │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────────┐   │
│  │  事实记忆   │ │  程序记忆   │ │    情境记忆     │   │
│  │  (Facts)    │ │ (Procedures)│ │   (Episodic)    │   │
│  └─────────────┘ └─────────────┘ └─────────────────┘   │
│  • 容量: 无限制 | 保留: 永久（需主动管理）              │
└─────────────────────────────────────────────────────────┘
```

#### 记忆类型

| 类型 | 示例 | 更新频率 |
|------|------|----------|
| **事实记忆** | 用户姓名、职位、技术栈、专业领域 | 低 |
| **程序记忆** | 工作流偏好、工具使用习惯、沟通模板 | 中 |
| **情境记忆** | 对话历史、项目经历、重要事件 | 高 |

#### 核心功能

| 功能 | 描述 |
|------|------|
| **记忆存储** | 自动提取和存储用户相关信息 |
| **语义检索** | 基于向量相似度的记忆检索 |
| **重要性评估** | 5维度评分模型(内容/反馈/访问/关联/时效) |
| **遗忘机制** | 基于艾宾浩斯遗忘曲线 + 重要性调整 |
| **记忆压缩** | 3级分层摘要策略 |
| **冲突解决** | 时间戳/重要性/合并/手动四种策略 |

#### 技术实现要点
- **向量数据库**: PostgreSQL + pgvector (P0/P1首选)，Milvus / Zilliz Cloud (大规模场景扩展)
- **嵌入模型**: OpenAI text-embedding-3-large (生产) / BGE-m3 (本地)
- **缓存**: Redis + 本地LRU
- **检索算法**: 混合检索(向量+关键词+上下文) + 重排序

### 2.3 ClawTeam代理团队模块

#### 功能概览
ClawTeam 在 v1.1 中定位为内部工作流执行引擎，而不是普通用户直接操作的主界面。平台保留多代理编排能力，但对外优先提供模板化场景、审批策略、执行追踪和结果复核。

#### 代理角色体系

```
┌─────────────────────────────────────────────────────────┐
│                    ClawTeam 角色体系                     │
├─────────────────────────────────────────────────────────┤
│  规划层                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │   Planner   │  │ Decomposer  │  │   Router    │     │
│  │   (规划者)   │  │  (分解者)   │  │  (路由者)   │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
├─────────────────────────────────────────────────────────┤
│  执行层                                                 │
│  ┌─────────────┐  ┌─────────────┐                      │
│  │   Executor  │  │  Specialist │                      │
│  │   (执行者)   │  │   (专家)    │                      │
│  └─────────────┘  └─────────────┘                      │
├─────────────────────────────────────────────────────────┤
│  验证层                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │  Validator  │  │   Critic    │  │ Synthesizer │     │
│  │  (验证者)   │  │  (批评者)   │  │  (综合者)   │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
└─────────────────────────────────────────────────────────┘
```

#### 团队拓扑模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **Hierarchical** | Manager-Worker结构 | 结构化任务 |
| **Peer-to-Peer** | 代理直接协作 | 探索性任务 |
| **Blackboard** | 共享工作空间 | 复杂问题求解 |
| **Pipeline** | 顺序处理 | 数据处理流程 |

#### 核心功能

| 功能 | 描述 |
|------|------|
| **工作流模板执行** | 以模板驱动场景执行，如需求拆解、状态汇总、风险检查 |
| **工具与连接器注册中心** | 统一管理外部工具、数据源和执行权限 |
| **任务分解与编排** | 将复杂任务拆为可追踪步骤，并按策略选择单代理或多代理执行 |
| **审批与回滚控制** | 对外部系统写入前创建审批请求，支持重试、撤销和人工接管 |
| **执行追踪与审计** | 记录每一步输入、输出、模型、工具调用、成本和耗时 |
| **MCP/工具协议兼容** | 兼容标准化工具协议，但对业务侧暴露稳定的产品化接口 |

#### 技术实现要点
- **代理框架**: CrewAI + LangGraph
- **LLM服务**: GPT-4o / Claude 3.5 Sonnet (主) + GPT-4o-mini (轻量)
- **消息队列**: RabbitMQ / Kafka
- **工具协议**: MCP (Model Context Protocol)

### 2.4 Azure DevOps Server集成模块

#### 功能概览
Azure DevOps 是平台最重要的外部事实源之一。设计目标不是复制一套项目管理系统，而是在保留 ADO 为 source of truth 的前提下，提供受控读取、增量同步、上下文增强和审批式写回。

#### 集成功能

| 模块 | 核心功能 |
|------|----------|
| **受控读取** | WIQL查询、工作项详情、仓库/PR、构建与发布状态、权限信息 |
| **上下文增强** | 生成状态摘要、阻塞项、风险提示、关联知识引用 |
| **审批式写回** | 创建工作项草稿、评论草稿、标签/字段建议，经确认后写入 |
| **同步治理** | 增量同步、游标管理、重试策略、幂等写入、审计日志 |
| **权限映射** | 继承企业身份体系和 ADO 权限边界，避免越权自动化 |

#### 身份验证方案

| 方式 | 适用场景 | 推荐度 |
|------|----------|--------|
| **PAT** | Azure DevOps Server/自动化脚本 | ⭐⭐⭐⭐ |
| **OAuth 2.0** | Azure DevOps Services/第三方应用 | ⭐⭐⭐⭐⭐ |
| **NTLM/Kerberos** | 企业内网/域环境 | ⭐⭐⭐⭐ |

#### 数据同步策略

```
┌─────────────────────────────────────────────────────────┐
│                    混合数据同步架构                       │
├─────────────────────────────────────────────────────────┤
│  实时同步 (Webhook/Service Hook)                        │
│  • 工作项变更 • 构建状态 • Pull Request                 │
├─────────────────────────────────────────────────────────┤
│  定时同步 (Quartz.NET)                                  │
│  • 工作项: 每5分钟 • 代码提交: 每10分钟                 │
│  • 构建状态: 每2分钟                                    │
├─────────────────────────────────────────────────────────┤
│  增量同步 (基于ChangedDate/ContinuationToken)           │
│  • 避免全量同步 • 优化API调用                           │
└─────────────────────────────────────────────────────────┘
```

#### 技术实现要点
- **API客户端**: Microsoft.TeamFoundationServer.Client / HttpClient + Polly
- **重试策略**: 指数退避 + 抖动
- **限流处理**: Token Bucket + Retry-After头解析
- **冲突解决**: 版本检测 + 字段级合并

### 3.2 模块间交互

#### 交互关系

| 交互 | 描述 |
|------|------|
| **外部系统 → Context Service** | 通过连接器、Webhook 和定时增量同步，把 ADO/文档/Git 数据归一化为可检索上下文 |
| **Knowledge ↔ Memory** | 知识内容可沉淀为长期记忆，记忆检索结果反向增强问答与工作流执行 |
| **Workflow Orchestrator ↔ ClawTeam** | 工作流编排器根据模板选择是否调用单代理或多代理执行，而非默认暴露原始代理配置 |
| **Workflow ↔ Approval/Writeback** | 涉及创建或修改外部系统数据时，先生成草稿和审批请求，再执行受控写回 |
| **Governance ↔ 全部模块** | 权限、审计、评测、提示词版本和成本控制作用于检索、执行和写回全链路 |

### 3.3 数据流设计

```
1. 外部系统与用户输入进入平台
   - 文档、工作项、仓库事件、聊天输入、会议文本
2. Context Service 做标准化与分层
   - source metadata、权限边界、结构化字段、向量索引、摘要缓存
3. Workflow Orchestrator 根据模板组装上下文
   - 检索知识、记忆、项目状态、历史执行记录
4. Execution Engine 执行任务
   - 单代理或多代理、工具调用、规则判断、生成草稿
5. Approval / Writeback / Audit
   - 受控写回外部系统，记录审批、执行轨迹、成本与结果
```

## 4. 技术栈建议

### 4.1 整体技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **后端** | Python + FastAPI | 高性能异步框架 |
| **前端** | React 18 + TypeScript + TipTap | 类型安全、组件化 |
| **桌面端** | Electron / Tauri | 本地优先体验 |
| **数据库** | PostgreSQL + pgvector | 结构化数据、JSONB文档和向量统一存储 |
| **缓存** | Redis | 会话、热点数据 |
| **消息队列** | RabbitMQ | 异步任务、事件驱动 |
| **对象存储** | MinIO / S3 | 附件、归档、导入导出文件 |
| **AI/LLM** | OpenAI / Azure OpenAI + LangGraph | 工作流执行与模型集成 |
| **部署** | Docker + Kubernetes | 容器化、可扩展 |

### 4.2 模块技术栈详细

#### 知识管理模块

| 组件 | 推荐方案 | 备选方案 |
|------|----------|----------|
| 编辑器核心 | ProseMirror + Tiptap | Slate.js |

## 5. 实施路线图

### 5.1 阶段规划

路线图以“闭环场景上线并稳定运行”为里程碑，而不是以“模块开发完成”为里程碑。每个阶段都必须同时满足业务价值、治理要求和运维可控性。

| 阶段 | 建议周期 | 目标 | 关键交付 |
|------|----------|------|----------|
| **Phase 0: 治理与底座基线** | 4-6周 | 建立可上线的最小生产底座 | 统一身份认证、RBAC、连接器框架、上下文标准模型、Prompt Registry、审计日志、成本统计、只读检索闭环 |
| **Phase 1: P0工作流闭环** | 6-8周 | 交付首批高频场景并验证效率收益 | 需求拆解到工作项草稿、迭代日报/周报生成、新人项目问答；支持来源引用、执行追踪、人工确认 |
| **Phase 2: 受控写回与审批** | 6-8周 | 让系统从“会建议”进化到“可落地执行” | 审批流、写回任务队列、回滚策略、幂等写入、会议纪要转行动项、发布风险检查 |
| **Phase 3: 平台化扩展** | 持续 | 把能力沉淀为团队级可复用平台 | 工作流模板中心、连接器扩展机制、评测平台、组织级运营后台、多团队隔离与配额管理 |

### 5.2 详细时间线

| 阶段 | 时长 | 核心交付 | 验收里程碑 |
|------|------|----------|------------|
| **Phase 0** | 4-6周 | 底座与治理基线 | 至少打通 1 个 ADO 项目与 1 个知识源；所有读取链路可审计 |
| **Phase 1** | 6-8周 | 3 个 P0 工作流闭环 | 核心场景端到端成功率达标，输出均带来源引用，人工复核路径清晰 |
| **Phase 2** | 6-8周 | 审批与受控写回 | 所有写操作先审批后执行，失败可重试，可追踪，可回滚 |
| **Phase 3** | 持续 | 平台化与规模化能力 | 新团队可通过模板与连接器复用能力，运营侧可观察成本、质量和使用情况 |

### 5.3 团队配置建议

| 角色 | Phase 0-1 | Phase 2 | Phase 3 |
|------|-----------|---------|---------|
| 平台/后端工程师 | 2 | 3 | 3 |
| 前端/工作台工程师 | 1 | 2 | 2 |
| AI应用工程师 | 1 | 1 | 2 |
| 企业集成/连接器工程师 | 1 | 1 | 2 |
| SRE/安全治理工程师 | 1 | 1 | 1 |
| 产品经理 | 1 | 1 | 1 |
| 设计师 | 1 | 1 | 1 |
| **总计** | **8** | **10** | **12** |

建议以“小而完整”的跨职能团队启动，不要先按模块拆成若干孤立小组，否则容易做出一堆能力碎片，却没有真正可推广的业务闭环。

## 6. 风险评估与应对

### 6.1 技术风险

| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| **AI成本控制** | 高 | 高 | 分层模型、语义缓存、Token优化 |
| **向量检索准确性** | 高 | 中 | 混合检索、重排序、持续调优 |
| **Azure DevOps API限流** | 中 | 高 | 增量同步、批量处理、缓存策略 |
| **大规模性能** | 高 | 中 | 分页、索引优化、缓存、CDN |

### 6.2 业务风险

| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| **用户接受度** | 高 | 中 | 渐进式功能、用户反馈循环 |
| **数据安全** | 高 | 低 | 加密、审计、合规认证 |
| **竞品压力** | 中 | 高 | 差异化功能、垂直深耕 |

### 6.3 非功能性需求

| 维度 | 目标 | 设计要求 |
|------|------|----------|
| **性能** | 常规读取 API P95 < 500ms；上下文组装 P95 < 2s；长工作流支持异步化 | 热点缓存、分层检索、异步任务队列、摘要缓存、增量同步 |
| **安全** | 不越权读取、不裸露密钥、不无审批写回 | 企业 SSO/OIDC、RBAC、密钥托管、字段脱敏、传输与存储加密 |
| **治理** | 所有模型调用和写操作可追踪、可回溯、可解释 | 审计日志、Prompt 版本管理、审批流、评测集、策略引擎、连接器权限边界 |
| **可靠性** | 核心读场景可用性目标 99.9%，写回任务具备幂等和补偿 | 重试策略、死信队列、任务状态机、幂等键、降级路径、人工接管 |
| **可扩展性** | 新工作流与新连接器能以模板方式复用，而非硬编码复制 | 统一工作流定义、连接器 SDK、工具注册中心、标准上下文接口 |
| **可维护性** | 关键链路易测试、易观测、易变更 | 契约测试、回放测试、指标/日志/Tracing、灰度发布、Schema 迁移规范 |
| **成本控制** | 单工作流成本可统计、可限额、可优化 | 模型分级路由、语义缓存、预算与配额、成本报表、异常调用告警 |
| **数据生命周期** | 记忆和上下文数据可设置保留、过期、归档和删除策略 | 数据分级、TTL、人工纠偏、删除审计、导出与清理机制 |
