# OKK 仓库执行约定

## 响应语言

- 默认使用中文简体回复。

## 执行方式

- 对于实现类任务，默认持续工作直到任务真正完成，不要做阶段性汇报。
- 用户一旦要求“继续”“继续做”“持续执行”“直接做完”，视为已经授权你自行串行推进后续多个 change；除非出现真实阻塞、危险操作确认点、或用户改变目标，否则不要停下来等待下一次“继续”。
- 除非满足以下条件，否则不要中途停下来总结进度：
  - 用户明确要求汇报
  - 出现真实阻塞，无法继续
  - 需要确认危险操作
- 工具调用前的说明压缩到最低；如非必要，不要为了“同步进度”而额外输出。
- 用户要求“持续执行直到全部完成”时，该要求高于默认协作式汇报风格。
- 完成一个 change 后，直接继续下一个 change，不要做过程性播报。
- 如果已经完成一个 change，但还能直接识别下一个高价值 change，则立刻进入下一轮 计划 - 执行 - 测试，不要用“是否继续”“目前做到这里”等话术把控制权交还给用户。
- 每个 change 都按照 计划 - 执行 - 测试 完整执行
- 完成一个 change 后，git commit 和 push。
- `git commit`、`git push`、测试通过、阶段目标达成，都只能视为一个中间里程碑，不能默认视为本轮任务结束。

## 完成定义

- 不要把“已经可用”、“已经可交付”、“主链路已打通”默认视为任务完成。
- 不要把“当前已经比之前更好”“设计稿主要部分已覆盖”“还剩一些优化项”视为可停止信号；只要剩余工作仍然明确提升交付质量，就继续做。
- 只有在以下条件同时成立时，才能停止并给出最终结论：
  - 用户请求的目标已经实现
  - 相关代码、脚本、文档、测试已经同步更新
  - 已完成必要验证，且没有明显残留的高优先级缺口
  - 当前继续推进不会再带来明确的质量提升或交付价值
- 如果仍存在明显的下一步高价值工作，应继续执行，而不是先停下来“汇报已完成”。
- 如果任务表述为“完整做完”“端到端交付”“直接按最优方案做”，则默认需要继续推进到生产可用边界，而不是只完成 MVP 主流程。

## 禁止过早收敛

- 不要因为通过了一轮测试、完成了一次提交、或者已经推送远端就停止。
- 不要把“我认为这已经够了”作为停止依据，停止依据只能来自用户目标、真实阻塞、危险操作确认点，或确实没有剩余高价值工作。
- 如果用户先前已经表达“继续做”或“直接做完”，则默认后续的优先级判断由你负责，不要在没有阻塞的情况下把“下一步做什么”重新抛回给用户。
- 不要把“需要整理下思路”“已经完成当前子任务”“我先汇报一下”当作暂停理由；这类行为在没有阻塞时都属于过早收敛。
- 当你判断“已经完成”时，必须先反向检查：
  - 是否还有未覆盖的关键链路
  - 是否还有明显缺失的工程化项
  - 是否还有可以直接继续完成且无需用户确认的工作
- 如果上述任一答案为“有”，则继续工作，不要结束。

## Git 与危险操作

- 涉及 `git reset --hard`、删除文件/目录、批量不可逆修改时，必须先明确确认。
