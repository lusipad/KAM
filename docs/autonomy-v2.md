# KAM Autonomy V2

## 目标

下一阶段的核心不是“再多一个工作台”，而是让 AI 能持续、反复地把同一个任务推进到可验收状态。

判断标准只有三组：

- 自主完成率
- 打断率
- 已完成工作的成功情况

## 设计原则

- 自治不是单次 run，而是一个 `Session -> Cycle -> Check` 的闭环。
- 完成必须以检查通过为准，不能只看模型输出“说自己完成了”。
- 打断必须被明确记录，不能混在失败态里。
- dogfooding 不是附加玩法，而是默认验证路径：先让 AI 在 KAM 仓库里持续开发 KAM 自己。

## 新对象

### `AutonomySession`

表示一次自治任务。

保存：

- 绑定任务
- 主 Agent
- 最大迭代次数
- 成功标准
- 检查命令
- 打断次数
- 最终状态

### `AutonomyCycle`

表示自治会话中的一轮迭代。

保存：

- 第几轮
- 对应的 worker run
- 检查结果
- 失败反馈
- 本轮状态

## 闭环流程

1. 选中一个任务，创建自治会话。
2. 系统自动生成 context，并拉起 worker run。
3. worker run 结束后，执行检查命令。
4. 如果检查通过，会话记为 `completed`。
5. 如果检查失败，系统把失败反馈拼进下一轮 prompt appendix，再次拉起 worker run。
6. 如果用户手动打断，会话记为 `interrupted`，并计入打断率。
7. 如果达到最大轮次仍未通过，会话记为 `failed`。

## 指标口径

### 自主完成率

`completed 且 interruption_count == 0` 的会话数 / 全部终态会话数

回答的问题：

- AI 是否能在不中断用户的前提下自己做完

### 打断率

`interrupted` 的会话数 / 全部终态会话数

回答的问题：

- 用户是否经常需要中途介入、暂停、重定向

### 完成成功率

`completed` 的会话数 / 全部终态会话数

回答的问题：

- 一旦会话结束，最终有多少是真的把工作做成了

## Dogfooding 模板

当前内置了 `KAM Dogfood` 模板，默认绑定本仓库，并自动执行：

- `app` lint
- `app` build
- `app` e2e
- `backend` unit tests

它的用途不是演示，而是把 KAM 的开发过程本身变成 KAM 的自治样本。

## 下一轮建议

- 给自治会话增加“验收 Agent”，把规则检查和语义检查分开
- 支持会话级 prompt 策略切换，例如修复优先 / 探索优先
- 把检查结果沉淀成可复用的任务模板
- 提供样本导出，用于分析哪些任务最容易打断、哪些检查最常失败
