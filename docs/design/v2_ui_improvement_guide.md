# KAM v2 — 界面与功能改进指南

> 本文档基于当前已实现的界面截图，给出具体的调整方向。
> 每条改动标注优先级：P0（必须立即做）、P1（本轮完成）、P2（下一轮）。
> 标注 🔴 删除 / 🟡 修改 / 🟢 新增。

---

## 一、当前状态问题总结

对照截图，当前界面有以下核心问题：

### 问题 1：空状态信息过载

界面上没有任何数据，但用户一眼能看到的信息点超过 20 个：顶栏标题 + 副标题、左侧 Projects 标题 + 副标题 + 输入框 + 按钮 + 说明文字、左侧 Threads 标题 + 副标题 + 数字 + 输入框 + 按钮、左侧底部三个导航项（PROJECTS / THREADS / RUNS）+ LOOK 按钮、中间四个零值统计（0 Runs / 0 Compare Sessions / 0 Pinned Resources / 0 Preferences）、中间大段引导文字、右侧 Context Panel 标题 + 副标题 + 四个 tab + 空状态提示、底部状态栏、输入区域旁边的 Agent 选择 + 勾选框 + 对比按钮 + 发送按钮。

**用户第一反应不是"我该干什么"，而是"这里有好多东西我看不懂"。**

### 问题 2：仍然是表单驱动

左侧有两个输入框（"新项目标题" + "新线程标题"），右侧有 Agent 下拉 + 勾选框。这要求用户先理解 Project / Thread / Run / Agent 四个概念的关系，然后手动操作。这和 v1 的"先创建任务卡再添加引用"本质相同。

### 问题 3：概念暴露过多

界面上同时出现了：Project、Thread、Run、Memory、Compare Sessions、Pinned Resources、Preferences、Context Panel、Agent、Command、Look。用户需要理解 11 个概念才能开始使用。v2 的目标是把用户可见概念砍到 3 个（Project / Thread / Run）。

### 问题 4：注意力分散

输入区本应是用户唯一的焦点，但它被 Agent 下拉、"发送时自动创建 Run"勾选框、"带去对比"按钮、"开始新项目"按钮、底部说明文字分割成了 6 个需要决策的区域。

## 二、全局导航栏（最左侧竖栏）

### 现状

```
[KAM 图标]
KAM
COMMAND

   ...

PROJECTS (橙点)
THREADS  (橙点)
RUNS     (橙点)

[LOOK 图标]
LOOK
```

### 调整

| 元素 | 动作 | 优先级 | 说明 |
|------|------|--------|------|
| `COMMAND` 文字 | 🔴 删除 | P0 | 不属于用户需要理解的概念 |
| `PROJECTS` 导航 | 🔴 删除 | P0 | 项目列表已在左侧栏内，不需要二级导航 |
| `THREADS` 导航 | 🔴 删除 | P0 | 同上 |
| `RUNS` 导航 | 🔴 删除 | P0 | Run 在对话中内联展示，不需要独立入口 |
| 橙色圆点 | 🔴 删除 | P0 | 去掉导航项后圆点也没有意义 |
| `LOOK` 按钮 | 🔴 删除 | P0 | 如果是调试/预览功能，移到设置中 |
| KAM 图标 + 文字 | 🟡 保留但简化 | P0 | 只保留图标，去掉文字，作为 home 按钮 |

### 目标效果

最左侧竖栏要么完全去掉（把 KAM logo 放进左侧栏顶部），要么只保留一个极简的图标栏：

```
[KAM logo]     ← 点击回到空状态/主页
               ← 大段空白
[⚙ 设置]       ← 底部唯一入口
```

推荐方案：**直接去掉最左侧竖栏**，把 logo 整合进左侧栏顶部。减少一层嵌套。

### 关键行为

- **创建项目**：点击 "+ New project" 弹出一个轻量模态框（只有标题 + 可选描述 + 可选仓库路径），或者直接在空状态的主输入框说话时系统自动创建
- **创建线程**：在选中的项目内，在主输入框发送第一条消息时自动创建。不需要用户手动命名
- **切换项目**：点击项目名即可。切换后 Threads 区域刷新为该项目的线程列表
- **切换线程**：点击线程名即可。切换后主区域加载该线程的对话历史

## 五、主区域 — 空状态

### 现状

```
       [大图标]

直接开始对话或选择现有 Project / Thread

这一版已经切到 v2 心智：Project 管持续上下文，Thread 管连续对话。
你现在可以直接输入第一条消息，系统会自动创建 Project、Thread，并

```

### 调整

| 元素 | 动作 | 优先级 | 说明 |
|------|------|--------|------|
| 图标 | 🟡 保留但缩小 | P0 | 用一个简洁的小图标即可，不需要很大 |
| 标题文字 | 🟡 改为更自然的提问 | P0 | "直接开始对话或选择现有 Project / Thread" 太技术化 |

## 七、主区域 — 对话流（有内容时）

### 现状

对话流目前还没有内容展示，以下是目标设计规范。

### 消息展示

对话流中有三种元素交替出现：

#### 1. 用户消息

```
                                    ┌──────────────────┐
                                    │ 帮我把 token 刷新 │
                                    │ 逻辑写完          │
                                    └──────────────────┘
```

- 靠右对齐
- 背景色：`bg-secondary`（浅灰）
- 圆角：12px
- 内边距：10px 14px
- 文字：14px
- 最大宽度：80% 容器宽度

#### 2. AI 回复

```
[K] 我分析了现有代码，当前使用 session-based auth。
    
    重构方案：
    1. 替换 session store 为 JWT 签名
    2. 添加 refresh token rotation
    3. 实现 token revocation

    开始实现第一步。
```

- 靠左对齐
- 无背景色（直接在白底上）
- 左侧有头像圆圈（`K` 或 KAM logo）
- 内容区支持 Markdown 渲染
- 最大宽度：85% 容器宽度

#### 3. Run 卡片（内联在对话流中）

Run 卡片是对话流的一部分，不是独立页面。根据 Run 状态展示不同样式：

**Pending 状态**

```
┌─ Run #1 ──────────────────────────── claude-code ─┐
│  ⏳ Pending                                        │
└───────────────────────────────────────────────────┘
```

- 细边框，灰色
- 一行信息足够

**Running 状态**

```
┌─ Run #1 ──────────────────────────── claude-code ─┐
│  🔵 Running... round 1/5          elapsed: 12s    │
│                                                    │
│  > Analyzing auth/middleware.py...                  │ ← 实时 stdout 最后几行
│  > Implementing refresh logic...                   │
│                                                    │
│  [查看完整日志]                      [取消]         │
└───────────────────────────────────────────────────┘
```

- 边框颜色：info 蓝色
- 状态指示器带呼吸动画
- 展示实时 stdout 最后 2-3 行
- 操作按钮：查看日志、取消

**Passed 状态**（最重要的状态，需要最丰富的信息）

```
┌─ Run #1 ──────────────────────────── claude-code ─┐
│  ✅ Passed · 3 rounds · 47s                        │
│                                                    │
│  实现了 token 刷新接口，添加了 race condition       │
│  保护。修改了 3 个文件，新增 2 个测试。             │
│                                                    │
│  auth/refresh.py     +45 -12                       │
│  auth/middleware.py   +8  -3                        │
│  tests/test_refresh.py +67 new                     │
│                                                    │
│  ✅ 14/14 tests passed                              │
│                                                    │
│  [查看 Diff]  [日志]  [采纳变更 ✓]  [重试]         │
└───────────────────────────────────────────────────┘
```

- 边框颜色：默认灰色（不用绿色，太刺眼）
- 状态行：绿色对勾 + 轮次 + 耗时
- Summary：2-3 行文字摘要
- 变更文件列表：文件名 + 增减行数，每个文件一个小胶囊标签
- 测试结果：一行 `✅ n/n tests passed`
- 操作按钮行：查看 Diff / 日志 / 采纳变更（主按钮）/ 重试

**Failed 状态**

```
┌─ Run #1 ──────────────────────────── claude-code ─┐
│  ❌ Failed · 5 rounds · 3m 20s                     │
│                                                    │
│  最大轮次已用完，测试仍未通过。                     │
│  最后一轮错误：TypeError: token.refresh is not...   │
│                                                    │
│  [查看日志]  [重试]                                 │
└───────────────────────────────────────────────────┘
```

- 边框颜色：`danger` 红色
- 展示最后一轮的错误信息，截断到 2 行

### Run 卡片的展开/收起

Run 卡片默认展示摘要视图（如上）。点击卡片或点击操作按钮可以展开为详情抽屉（从右侧滑入或在卡片下方展开），包含：

- **完整日志 tab**：每一轮的 stdout / stderr，按轮次分组
- **Diff tab**：完整的代码变更 diff 视图
- **检查结果 tab**：每一轮检查命令的输出

## 十、对比功能

### 现状

### 调整方案

**触发方式 1 — 对话触发**：用户说"分别用 Claude Code 和 Codex 实现这个功能"，Conversation Router 识别为 COMPARE 意图，同时创建两个 Run。两个 Run 卡片在对话中并排展示。

## 十二、颜色与视觉风格

### 现状

深色主题，以深灰/黑色为底，橙色为强调色。

### 建议

当前的深色主题可以保留，但需要注意几点：

1. **降低橙色的使用频率**。当前橙色出现在：图标、按钮、圆点、边框。过多的强调色会分散注意力。建议只在"主操作按钮（发送）"和"活跃状态指示"上使用橙色。
2. **Run 状态使用语义色**：
   - Running：蓝色（info）
   - Passed：绿色（success）
   - Failed：红色（danger）
   - Pending/Cancelled：灰色
3. **空状态的引导文字用更淡的灰色**，不要和正文一样的亮度。
4. **输入框的边框在聚焦时用橙色高亮**，非聚焦时用低对比度灰色边框。
5. **左侧栏和主区域要有明确的视觉层级**。左侧栏背景应该比主区域略深一个层级。

## 十四、关键交互流程参考

### 流程 A：全新用户首次使用

```
1. 用户打开 KAM
2. 看到空状态：图标 + "你在做什么？" + 输入框
3. 用户在输入框写："我要重构用户认证模块，改用 JWT"
4. 用户按 Cmd+Enter 发送
5. 系统自动：
   a. 创建 Project "重构用户认证"
   b. 创建 Thread（标题自动生成）
   c. 左侧栏刷新，出现新项目和新线程
   d. 面包屑更新为"重构用户认证 / 初始分析"
6. AI 回复出现在对话流中
7. 如果触发了 Run，Run 卡片内联出现在 AI 回复下方
```

### 流程 B：回到之前的工作

```
1. 用户打开 KAM
2. 左侧栏显示项目列表，上次活跃的项目在最上方
3. 用户点击项目名 → 线程列表展开
4. 用户点击线程名 → 主区域加载对话历史
5. 对话历史中已有的 Run 卡片显示最终状态
6. 用户在输入框继续对话
```

### 流程 C：对比多个 Agent

```
1. 用户在对话中说："分别用 Claude Code 和 Codex 实现这个功能"
2. 系统创建两个 Run，对话流中出现两个 Running 状态的 Run 卡片
3. 两个 Run 分别完成后，对话流中出现对比卡片
4. 用户查看对比结果，选择采纳其中一个
```

## 十五、给 AI Agent 的实现提示

### 删除操作优先

先做 P0 的删除操作。删代码永远比加代码容易，而且效果立竿见影。把上面标记 🔴 的元素全部从界面上移除，不需要重写组件，只需要注释掉或删除对应的 JSX。

### 不要做的事

- 不要引入新的 UI 库或组件库来实现这些改动
- 不要重写现有的后端 API（前端改完再动后端）
- 不要在删除的同时添加新功能（先减后加）
- 不要保留任何"以防万一"的空状态提示文字

### 一条检验标准

完成所有修改后，截图发给一个从没见过 KAM 的人，问他："你觉得这个界面让你干什么？"如果他的回答是"打字告诉它我要做什么"，就对了。如果他的回答是"我需要先搞清楚 Project、Thread、Run 是什么"，就还没改到位。
