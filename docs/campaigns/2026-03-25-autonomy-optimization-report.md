当前系统已经能用真实 `git worktree` + 真实检查命令做自治开发评估，并且可以根据失败样本反向改造系统本身。核心目标不再是“把更多页面堆出来”，而是持续提高三件事：

## 这轮系统改动

### 1. 修掉 Windows 取消清理竞争

提交：`2702245` `Fix Windows run cleanup race`

作用：

- 打断自治会话后，等待 run 线程真正收口
- 释放 `stdout/stderr` 日志句柄
- 避免后续测试或下一轮自治被文件锁卡住

### 2. 给 worktree 自动补运行时

提交：`9e1a37c` `Hydrate worktree runtime for autonomy checks`

作用：

- 在自治 worktree 中自动挂接 `app/node_modules`
- 在自治 worktree 中自动挂接仓库根目录 `.venv`
- 让 `dogfood` 检查真的跑在当前 worktree，而不是误跑主仓库
- 优化战役脚本和 dogfood 默认检查都优先使用稳定 Python 路径

这是本轮最关键的改动。基线战役里唯一失败样本，本质上不是任务理解错，而是 worktree 里缺少可执行运行时，导致 `eslint`、`tsc`、`playwright` 不可用。补齐这层后，第二轮 10 个任务全部完成。

## 是否偏离初衷

没有。

反而是把“做产品界面”重新拉回了“做自治开发系统”：

- 关注点从展示层扩展，收敛回了自治闭环
- 衡量方式从主观感受，收敛回了可重复战役与硬指标
- 改动方式从继续堆功能，收敛回了提升完成率与成功率的系统工程

如果按这条线继续推进，后续版本应该优先优化自治效率、失败画像与长期运行能力，而不是继续扩大非核心 UI 面积。
